TTFB Optimization

Guía práctica para reducir el Time to First Byte de 2.0s a < 0.8s en Cloudflare Pages.

Diagnóstico

El TTFB (Time to First Byte) mide cuánto tarda el servidor en responder el primer byte de datos. Para un portfolio estático en Cloudflare Pages, debería ser < 0.3s.

2.0s
Actual (TTFB)
< 0.3s
Objetivo
0.0s
CLS (actual)
100
SEO / A11Y
¿Por qué 2.0s es alto? Un sitio estático servido desde el edge de Cloudflare debería responder en milisegundos. Si tarda 2s, algo está bloqueando la respuesta.

Causas comunes

CausaSíntomaSolución
Edge cache miss Primer request lento, segundo rápido Cache-Control headers + pre-warm
Build output no optimizado TTFB lento siempre Verificar output: 'static'
SSL/TLS negotiation Lento solo en primera visita Cloudflare proxy + HSTS
DNS resolution lento TTFB + connection time alto Cloudflare DNS + preconnect
Functions/Workers activos TTFB alto incluso para HTML estático Desactivar SSR innecesario

Plan de corrección

1

Verificar modo static (no SSR)

Este portfolio es 100% estático. Asegúrate de que astro.config.mjs tenga:

astro.config.mjs 
export default defineConfig({
  output: 'static',  // ← CRÍTICO: genera HTML puro
  // NO usar output: 'server' si no necesitas SSR
})

Si usas output: 'server', Cloudflare ejecuta un Worker para cada request — eso añade 200-800ms de cold start.

2

Configurar Cloudflare Cache

En Cloudflare Dashboard > Caching > Configuration:

  • Caching Level: Standard
  • Browser Cache TTL: 1 year (para assets con hash)
  • Edge Cache TTL: 1 month (para HTML estático)
public/_headers (ya incluido) 
# Cache para assets estáticos (1 año)
/assets/*
  Cache-Control: public, max-age=31536000, immutable

# Cache para HTML (revalida en cada request)
/*.html
  Cache-Control: public, max-age=0, must-revalidate
3

Pre-warm el edge cache

Después de cada deploy, Cloudflare sirve HTML desde el Worker sin caché. Para evitar TTFB alto en la primera visita:

Tip: Usa Cloudflare's Cache Reserve o crea un Cloudflare Worker que haga un fetch() a tu URL después de cada deploy para calentar el edge cache. El script está incluido abajo.
4

Worker de Cache Warm (opcional)

Este Worker se ejecuta después de cada deploy para "calentar" el cache del edge:

wrangler.toml 
name = "cache-warm"
main = "cache-warm.js"
compatibility_date = "2026-04-01"

// Trigger después de cada deploy de Pages
[triggers]
crons = ["0 */6 * * *"]  // Cada 6 horas

[env.production]
vars = { SITE_URL = "https://buyportfolio.zutra.agency" }

[[routes]]
pattern = "buyportfolio.zutra.agency/*"
zone_name = "zutra.agency"
cache-warm.js 
// Cache warm — hace fetch de todas las rutas importantes
// para que el edge cache las tenga listas

const PAGES = [
  '/',
  '/projects',
  '/blog',
  '/services',
  '/contact',
  '/changelog',
];

export default {
  async scheduled(controller, env, ctx) {
    const results = await Promise.allSettled(
      PAGES.map(path => fetch(env.SITE_URL + path))
    );

    const warmed = results.filter(r => r.status === 'fulfilled');
    console.log(`Cache warm: ${warmed.length}/${PAGES.length} pages cached`);
  },
};
5

Reducir el tamaño del HTML

Un HTML más grande = más tiempo para el primer byte. Optimizaciones:

  • Eliminar astro:transitions si no lo usas (añade ~14 KB de JS al <head>)
  • Minificar HTML: Astro ya lo hace en build
  • Reducir JSON-LD: incluir solo en las páginas que lo necesitan
  • Eliminar <script> inline innecesarios del <head>
6

Verificar en producción

Después de aplicar los cambios, verifica:

Terminal 
# Test 1: Primera visita (sin cache)
curl -o /dev/null -w "TTFB: %{time_starttransfer}s\n" \
  -H "Cache-Control: no-cache" \
  https://buyportfolio.zutra.agency/

# Test 2: Segunda visita (con cache)
curl -o /dev/null -w "TTFB: %{time_starttransfer}s\n" \
  https://buyportfolio.zutra.agency/

# Test 3: Headers de cache
curl -I https://buyportfolio.zutra.agency/ | grep -i cache
